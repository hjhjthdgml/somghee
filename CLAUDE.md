# 개발 원칙 — 코딩 가이드

## ⚠️ 최우선 원칙: "모르고 깨져있는 게 가장 위험하다"

에러가 터지면 그나마 발견이 된다. 진짜 문제는 겉으로 정상처럼 보이는데 실은 작동 안 하는 코드다.

- ❌ 최악: 페이지 열림 + 데이터 0건 표시 + 에러 없음 → "데이터가 아직 없나보다" → 며칠 후 발견
- ❌ 위험: `.catch(() => ({docs:[]}))` → 에러를 삼켜서 빈 배열 반환 → 정상처럼 보임
- ❌ 위험: 잘못된 컬렉션명 → 빈 스냅샷 반환 → 에러 없이 빈 목록
- ✅ 올바른 접근: 에러는 반드시 표면에 드러나게. 삼키지 말 것.

---

## 1. 코딩 시작 전 — 필수 확인 (건너뛰기 금지)

### ① 기존 파일 읽기 (가장 먼저)
반드시 기존 배포 파일의 코드를 읽고 패턴을 파악한 후 코딩 시작.
"내 방식대로 새로 만들기" 금지.

### ② 코드베이스 표준 체크리스트
1. Firebase SDK 버전 — 임의 변경 금지
2. 인증 가드 포함 여부 확인
3. 자체 `firebase.initializeApp` 금지 — 인증 가드가 처리
4. DB/Auth 참조 변수 — 자체 선언 금지, 인증 가드가 제공하는 변수 사용
5. 초기화 패턴 통일
6. 사이드바 — 자체 하드코딩 금지
7. CSS 변수 체계 — 기존 `:root` 변수 복사, 새 체계 만들지 말 것
8. Firestore 쿼리 — `orderBy` 금지 → `.get()` 후 클라이언트 정렬 또는 try-catch 폴백

## 2. 코딩 중 — "조용한 실패" 방지 규칙

### 🔴 규칙 A: 에러를 삼키지 마라
```js
// ❌ 최악: 에러를 삼켜서 빈 배열 반환
collection.get().catch(() => ({docs:[]}));

// ✅ 올바름: 에러를 사용자에게 보여줌
try {
  const snap = await collection.get();
  renderList(snap.docs);
} catch (e) {
  console.error('[로드 실패]:', e);
  toast('데이터 로드 실패: ' + e.message, 'err');
}
```

### 🔴 규칙 B: 빈 결과는 의심하라
```js
const snap = await collection.get();
console.log(`${snap.docs.length}건 로드`); // 반드시 건수 확인 로그

if (snap.docs.length === 0) {
  renderEmpty('데이터가 없습니다');
  return;
}
```

### 🔴 규칙 C: 쿼리 안전 패턴
```js
// ✅ 안전 (인덱스 불필요)
const snap = await collection.get();
const sorted = snap.docs
  .map(d => ({id: d.id, ...d.data()}))
  .sort((a, b) => toTs(b.createdAt) - toTs(a.createdAt));

// ✅ 폴백 (어쩔 수 없이 orderBy 쓸 때)
try {
  snap = await collection.orderBy('createdAt','desc').limit(5).get();
} catch (e) {
  console.warn('orderBy 실패, 인덱스 없음 → 폴백:', e.message);
  snap = await collection.limit(5).get();
}
```

### 🔴 규칙 D: 파일 형식 규칙
- `.js` 파일 = 순수 JavaScript만
  - ❌ HTML 주석(`<!-- -->`), `<script>` 태그 절대 금지
  - ✅ 주석은 `/* */` 또는 `//` 만 사용
- `.html` 파일 내 JS 템플릿 리터럴:
  - ❌ 리터럴 안에 `</script>` 직접 쓰면 브라우저가 스크립트 종료로 인식
  - ✅ 반드시 `<\/script>`로 이스케이프

### 🔴 규칙 E: DOM 접근 타이밍
- `<head>`에서 로드되는 스크립트:
  - ❌ `document.body` 직접 접근 (body가 아직 없음 → null 에러 → 이후 코드 전부 중단)
  - ✅ `document.head`에 `<style>` 삽입 방식
  - ✅ 또는 `DOMContentLoaded` 이벤트 사용

### 🔴 규칙 F: 연쇄 의존성 인식
핵심 공통 파일이 깨지면 → 전역 변수 미정의 → 전체 사이트 다운.
핵심 파일 수정 시 반드시 다른 페이지도 함께 테스트.

### 🔴 규칙 G: DB 컬렉션명 오타 주의
Firestore는 존재하지 않는 컬렉션을 쿼리해도 에러가 아니라 빈 스냅샷을 반환한다.
→ 컬렉션명은 반드시 스키마 문서와 대조 확인. 복사-붙여넣기 권장.

### 🔴 규칙 H: 상태값 불일치 주의
status enum 값은 반드시 소문자 + 언더바/하이픈 없는 단어로 통일.
- 예: `"active"` ✅ / `"Active"` ❌ / `"is_active"` ❌

### 🔴 규칙 I: 새 페이지 생성 시 공통 CSS 누락 금지
새 페이지 필수 CSS 체크리스트:
- [ ] `.sidebar` (width 고정, fixed, left:0, height:100vh)
- [ ] `.sidebar .nav-item` (flex, padding, hover, active 상태)
- [ ] `.mobile-header` (display:none → @media 768px에서 flex)
- [ ] `.main-content` (margin-left 고정 → @media 768px에서 0)
- [ ] `@media (max-width:768px) .sidebar { display:none }`

★ 가장 안전한 방법: 기존 작동하는 페이지에서 `<style>` 블록 전체를 복사 후 페이지별 CSS만 추가

### 🔴 규칙 J: 파생 필드 동기화 주의
파생값(계산으로 구할 수 있는 값)을 DB에 별도 저장할 때는 반드시 원본과 동시 업데이트.
→ 현재 단계에서는 저장하지 않고 클라이언트 계산으로 처리 권장.

## 3. 코딩 완료 후 — 납품 전 검증

### 기본 5대 체크
1. 기능 작동 — 모든 버튼·입력·저장·조회가 실제로 동작하는가?
2. 논리 정합성 — 데이터가 처음부터 끝까지 맞아떨어지는가?
3. 실사용 시나리오 — "등록→수정→조회→삭제"를 막힘 없이 할 수 있는가?
4. 빈 상태/에러 — 데이터 없을 때, 네트워크 끊길 때, 잘못 입력할 때 안 깨지는가?
5. 기존 호환 — 다른 페이지와 함께 배포했을 때 충돌 없는가?

### ★ "조용한 실패" 전용 체크
6. 콘솔 에러 0건인가? — F12 → Console 확인. 경고(노란색)도 확인
7. 데이터가 실제로 보이는가? — 실제 DB 데이터가 화면에 표시되는가?
8. "빈 결과"가 진짜 빈 건가? — DB에 데이터 있는데 0건이면 쿼리/컬렉션명/상태값 오류
9. 다른 페이지 안 깨졌는가? — 공통 파일 수정 시 다른 페이지도 정상 확인
10. 모바일에서도 되는가? — 반응형 레이아웃, 터치 동작 확인
11. 파생 필드 동기화 맞는가? — 원본 필드 변경 후 화면 표시 파생값이 맞는지 확인

## 4. "조용한 실패" 유형 사전 목록

| 유형 | 증상 | 원인 | 방지 |
|---|---|---|---|
| A. 쿼리 0건 | 목록 비어있음, 에러 없음 | 컬렉션명 오타, 인덱스 없음, status 대소문자 불일치 | console.log 건수 + 스키마 대조 |
| B. 에러 삼킴 | 정상처럼 보임 | `.catch(() => 빈값)`, 빈 catch 블록 | catch에서 반드시 toast + console.error |
| C. 비동기 순서 | 가끔 데이터 안 나옴 | await 빠뜨림, 의존 데이터 미로드 | async/await 철저히. 의존 데이터는 순차 로드 |
| D. DOM 미존재 | 특정 기능만 안 됨 | getElementById 대상이 아직 렌더링 안 됨 | DOM 조작은 렌더 이후에 |
| E. 이벤트 중복 | 버튼 2번 실행, 데이터 2배 저장 | addEventListener 중복 등록 | onclick 속성 사용 또는 등록 전 제거 |
| F. CSS 덮어쓰기 | 레이아웃 미묘하게 깨짐 | 같은 클래스명 충돌 | 기존 CSS 변수 사용 |
| G. Timestamp 혼재 | 정렬 안 됨, 날짜 이상 | DB Timestamp vs ISO string 혼재 | toTs() 유틸로 통일 변환 |
| H. 배열 필드 쿼리 | where로 못 찾음 | 배열 필드는 DB 쿼리 제한 | 클라이언트에서 필터링 |
| I. 권한 에러 | 저장 안 됨 | DB Rules에서 차단 | 개발 중 콘솔 항시 확인 |
| J. 캐시 문제 | 수정했는데 안 바뀜 | 브라우저/CDN 캐시 | Ctrl+Shift+R 강력 새로고침 |
| K. 공통 CSS 누락 | 사이드바 깨짐 | 새 페이지에서 CSS 누락 | 기존 페이지 style 블록 전체 복사 후 시작 |
| L. 파생 필드 불일치 | 표시 수치 오류, 에러 없음 | 원본 필드 업데이트 후 파생 필드 미갱신 | 파생 필드는 클라이언트 계산 or 트랜잭션 처리 |

## 5. 배포 및 운영 규칙

**배포:**
1. 파일 업로드 후 반영까지 1~5분 소요
2. 강력 새로고침: Ctrl+Shift+R
3. 공통 파일 수정 시 → 모든 페이지 영향. 의존 순서대로 업로드

**모바일 대응 필수:**
- `@media (max-width: 768px)` 기본 포함
- 버튼 최소 44px × 44px (터치 영역)

**데이터 보호:**
1. 삭제 전 반드시 `confirm()` 확인
2. 실시간 리스너(onSnapshot) 내에서 write 금지 (무한 루프 위험)
3. 저장 시: 서버 타임스탬프 사용 권장 (브라우저 시간 금지)

**파일 간 복사 시 체크:**
- [ ] 페이지 title 변경
- [ ] 사이드바 메뉴 ID 변경
- [ ] 컬렉션명이 이전 페이지 것 그대로 아닌지 확인
- [ ] 변수명/함수명 충돌 없는지 확인

## 6. 스키마 설계 원칙

1. **Single Write Multi Read**: 데이터는 한 곳에만 쓰고 여러 곳에서 읽는다. 중복 저장 금지.

2. **가격 레이어 분리:**
   - 상품 마스터 (기준가 + MOQ 단가표)
   - 판매 레이어 (실제 판매가)
   - 계약 레이어 (당사자별 수수료/특이사항)
   - 하위 레이어는 상위 레이어를 참조만 하고 덮어쓰지 않는다.

3. **status enum 값은 반드시 소문자 단어.**
   예: `"active"` ✅ / `"Active"` ❌ / `"is_active"` ❌

4. **Timestamp 저장 규칙:**
   - 생성/수정 시각: 서버 타임스탬프 사용 (브라우저 시간 금지)
   - 필드명: `createdAt` / `updatedAt` / `[이벤트명]At`

5. **ID 참조 규칙:**
   - 외래키 참조 시 컬렉션명 + "Id" suffix 통일
   - 참조 무결성은 앱 레이어에서 보장

6. **파생 필드는 원칙적으로 DB에 저장하지 않는다.**
   화면에서 실시간 계산. 단, 트랜잭션으로 원자적 처리가 보장될 때만 저장 허용.

7. **모든 금전 이동은 별도 트랜잭션 컬렉션에 건별로 기록.** 세무 근거 역할.

8. **접근 권한 계층을 명확히 설계**하고, 민감 데이터는 관리자 권한만 접근.
